# PR: Phase 1.1.c — trips as source of truth in loadPersistedPlannerState

## Scope

Promueve la tabla `trips` a source of truth para la carga de estado del planner. `loadPersistedPlannerState` ahora lee desde `trips.planner_state` (escrito por `upsertTrip` desde 1.1.b) y cae a `messages` solo cuando no existe un trip row (conversaciones pre-1.1.b). Continua [Fase 1.1.b (#61)](1.1.b-upsert-trip-adapter.md).

## Que cambia

### `src/features/trip-planner/hooks/usePlannerState.ts`

- `loadPersistedPlannerState` refactorizado: el paso 2 (source of truth) ahora llama a `getTripByConversation(conversationId)` primero. Si retorna `TripPlannerState`, lo normaliza y lo usa. Si retorna `null` (no existe trip row), cae al query existente contra `messages`.
- Error handling: si `getTripByConversation` tira excepcion (network, RLS, etc.), loguea con `console.error` y cae al fallback de messages sin romper UX.
- Telemetria debug-level en los 3 caminos de lectura: `indexeddb cache`, `trips`, `messages (fallback)`. Efimera, se remueve en 1.1.g.
- Import de `getTripByConversation` agregado desde `tripService`.

### `src/features/trip-planner/services/tripService.ts`

- `getTripByConversation` corregido:
  - `single()` reemplazado por `maybeSingle()` — evita error PGRST116 cuando no hay trip row (esperado para conversaciones pre-1.1.b).
  - Error de Supabase ahora se propaga via `throw` para que el caller pueda distinguir "no trip" (null) de "query rota" (excepcion).

## Flujo resultante de loadPersistedPlannerState

```
1. IndexedDB cache → display instantaneo (sin cambio)
2. getTripByConversation(conversationId)
   ├─ TripPlannerState → normalizar, usar, actualizar cache → FIN
   ├─ null → continuar a paso 3
   └─ throw → log error, continuar a paso 3
3. Fallback: query a messages (role='system', meta.messageType='trip_planner_state')
   ├─ encontrado → normalizar, usar, actualizar cache → FIN
   └─ null → setPlannerState(null) → FIN
```

  1. `getTripByConversation` retorna state cuando trip existe
  2. `getTripByConversation` retorna null cuando no hay trip row (pre-1.1.b)
  3. `getTripByConversation` tira excepcion en error de Supabase
  4. `normalizePlannerState` produce output identico desde datos de trip y messages
  5. `normalizePlannerState` produce `TripPlannerState` valido con todos los campos requeridos
- Baseline pre-existente: **132 passed** sin cambios. Los 2 failed (D11 localStorage) siguen igual.

## Backwards compatibility

- **Agent flow identico**: `getTripByConversation` devuelve el mismo `TripPlannerState` que hoy viene de messages, normalizado por la misma funcion (`normalizePlannerState`). Ningun test pre-existente cambia resultado.
- **Dual-write preservado**: `persistPlannerState` no se toca. Sigue escribiendo a messages y a trips (throttled 5s). Se elimina en 1.1.g.
- **Conversaciones pre-1.1.b**: fallback transparente a messages. Sin backfill on-the-fly.

## Que NO se toco

`persistPlannerState`, throttle 5s (1.1.d), `listTripsByUser` (1.1.e), `deriveTripStatus` (1.1.f), dual-write a messages (1.1.g), UI/rutas/layouts, orchestrator, `useMessageHandler`, `planner_agent`, `duplicateTrip` (D15).

## Riesgos

- **R1 — Throttle 5s puede causar lag en primera lectura**: Si el usuario persiste un estado y navega a otra conversacion y vuelve antes de 5s, el trip row puede no existir todavia. El fallback a messages cubre este caso. Permanente hasta 1.1.d reemplace el throttle por debounce+flush.
- **R2 — `getTripByConversation` y RLS consumer**: Las RLS policies de 1.1.a (`consumer_select_own_trips`) exigen `owner_user_id = auth.uid()`. Si un consumer intenta leer un trip de otra persona, el query devuelve null (no error) y cae a messages. Correcto.
