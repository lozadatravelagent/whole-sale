# PR: Phase 1.1.b — upsertTrip Adapter + usePlannerState Guard

## Scope

Adapter en `tripService.upsertTrip` y guard en `usePlannerState` para habilitar persistencia de trips de consumers B2C, manteniendo el comportamiento B2B identico. Continua [Fase 1.1.a (#60)](1.1.a-b2c-ownership.md) y sigue el plan del [handoff de cierre de prerrequisitos](../handoffs/Handoff%20%E2%80%94%20Emilia%20B2C%2C%20estado%20al%20cierre%20de%20prerrequisitos%20de%20Fase%201.1.b.md).

## Que cambia

### `src/contexts/AuthContext.tsx`

- `AuthUser` expone `accountType: 'agent' | 'consumer'`.
- Query a `public.users` ahora incluye `account_type`.
- Safety net `|| 'agent'` para users de prod sin la columna (D12 pendiente).

### `src/features/trip-planner/services/tripService.ts`

- Nueva signature de `upsertTrip`: `agencyId`/`tenantId` pasan a `string | null`, nuevo param `accountType: 'agent' | 'consumer' = 'agent'`.
- Validaciones pre-DB: agent exige `agencyId`+`tenantId`, consumer exige `userId`.
- `tripData` ahora incluye `owner_user_id` (siempre) y `account_type`. Para consumer, `agency_id`/`tenant_id` se nulean explicitamente.

### `src/features/trip-planner/hooks/usePlannerState.ts`

- Guard de `persistPlannerState` extendido: agents siguen exigiendo `agency_id`+`tenant_id`, consumers pasan sin esos campos. Llamada a `upsertTrip` pasa `accountType` derivado del `AuthUser`.

## Tests

- **6 unit tests nuevos** en `upsertTripAdapter.test.ts` (mock de Supabase): agent default, agent explicito, consumer valido, agent sin agency (falla pre-DB), consumer con agency residual (se nulea), y `accountType` undefined defaulteando a agent (simula prod sin D12).
- **2 integration tests nuevos** en `b2cOwnershipRls.test.ts`: consumer persiste trip via insert end-to-end (Test 8), agent persiste trip con `owner_user_id` + `account_type='agent'` regression (Test 9).
- Baseline pre-existente: **126 passed** sin cambios. Los 2 failed pre-existentes (D11 localStorage) siguen igual.

## Backwards compatibility

- Default `accountType='agent'` → el unico call site existente (`usePlannerState.ts:154`) funciona identico sin modificar su llamada conceptualmente.
- `|| 'agent'` en AuthContext → el codigo corre contra prod aunque D12 no este aplicada.
- Unico call site modificado de `upsertTrip` en todo el codebase: `usePlannerState.ts`. Los demas usos de `tripService` (`listTripsByAgency`, `updateTripLeadId`, `TripRow` como tipo) quedan intactos.

## Que NO se toco

`deriveTripStatus`, `loadPersistedPlannerState` (1.1.c), dual-write a messages (1.1.g), throttle 5s (1.1.d), `listTripsByUser` (1.1.e), `duplicateTrip` (ver D15), UI/rutas/layouts, orchestrator, `useMessageHandler`, `planner_agent`, `Role` type (no agrega `'CONSUMER'`; es preocupacion de auth/routing B2C futuro).

## Riesgos

- **R1 → D15:** `duplicateTrip` tiene el mismo bug de `owner_user_id` faltante. Sin call sites actuales. Documentado en `TECH_DEBT.md` como D15, severidad baja, fix trivial. No se arregla en esta PR.
- **R2 mitigado:** AuthContext lee de tabla con default `'agent'` si la columna no existe.
- **R3, R4:** fuera de scope (login consumer aun no implementado; `canPersistAgent` con `undefined` se comporta como agent, que es el comportamiento actual).

## Bloqueante para deploy — D12

**Esta PR no puede deployarse a produccion hasta que las migrations de 1.1.a (`20260409000001_add_consumer_role_value.sql` y `20260409000002_b2c_ownership.sql`) esten aplicadas al remoto.** El codigo depende de `trips.owner_user_id`, `trips.account_type`, `public.users.account_type` y las RLS policies `consumer_*`. Ver D12 en `TECH_DEBT.md` para el checklist de push a prod.

Mergear a main es seguro (main no se deployea automaticamente), pero cualquier release debe resolver D12 primero.
