# PR: Phase 1.1.d ‚Äî Replace trip upsert throttle with debounce + flush

## Scope

Reemplaza el throttle manual de 5s en `persistPlannerState` por un debounce trailing de 3s con flush garantizado en unmount, beforeunload y cambio de conversacion. Esto cierra la ventana de perdida de datos donde edits rapidos + cierre de browser dejaban la tabla `trips` desactualizada. Continua [Fase 1.1.c](1.1.c-trips-source-of-truth.md).

## Problema que resuelve

Desde 1.1.c, `loadPersistedPlannerState` lee de `trips` como source of truth. El throttle de 5s en `upsertTrip` significaba que si el usuario editaba rapidamente y cerraba el browser dentro de la ventana de 5s, el ultimo estado se perdia en `trips` (aunque quedaba en IndexedDB y messages). Con debounce + flush, el write pendiente se ejecuta antes de que el componente se desmonte o el browser se cierre.

## Que cambia

### `src/features/trip-planner/hooks/createDebouncedFlusher.ts` (nuevo)

- Funcion pura `createDebouncedFlusher(delayMs)` que retorna `{ schedule, flush, cancel }`.
- Trailing debounce: multiples `schedule()` rapidos colapsan en una sola ejecucion despues del delay.
- `flush()` ejecuta el callback pendiente inmediatamente y cancela el timer. No-op si no hay pendiente.
- `cancel()` descarta sin ejecutar.

### `src/features/trip-planner/hooks/usePlannerState.ts`

- `lastTripUpsertRef = useRef(0)` reemplazado por `tripUpsertDebounceRef = useRef(createDebouncedFlusher(3000))`.
- Bloque throttle (antiguo `now - lastTripUpsertRef.current > 5000`) reemplazado por `tripUpsertDebounceRef.current.schedule(...)`. El `canPersist` guard se mantiene identico.
- Nuevo `useEffect` (lineas 98-107): registra handler `beforeunload` que flushea el debounce; cleanup del effect flushea en unmount (cambio de ruta).
- Flush explicito en bloque de cambio de conversacion (linea 64): si hay upsert pendiente para la conversacion anterior, se ejecuta antes del reset de estado.
- Import de `createDebouncedFlusher` agregado.

## Que NO se toco

- **Messages dual-write** (delete+insert en cada `persistPlannerState` call) ‚Äî sin cambio, se elimina en 1.1.g.
- **IndexedDB cache** (`setPlannerStateInCache`) ‚Äî sin cambio, sigue fire-and-forget en cada call.
- **`loadPersistedPlannerState`** ‚Äî cerrado en 1.1.c, sin modificacion.
- **`beforeunload` existente** (lineas 88-96) ‚Äî sigue mostrando warning UX para mutations activas. Concern independiente.
- UI, componentes, rutas, layouts, orchestrator, `useMessageHandler`, `planner_agent`.
- `listTripsByUser` (1.1.e), `deriveTripStatus` (1.1.f), `duplicateTrip` (D15).

## Tests

- **7 unit tests nuevos** en `tripUpsertDebounce.test.ts`:
  1. Debounce agrupa multiples llamadas rapidas en un solo fire
  2. `flush()` dispara el write pendiente inmediatamente
  3. `flush()` es no-op sin pendiente
  4. `cancel()` descarta sin ejecutar
  5. Timer se reinicia en cada schedule
  6. No double-fire despues de flush + timer expiry
  7. Flush despues de fire natural es no-op
- Usa `vi.useFakeTimers()` para control preciso del timing.
- Baseline pre-existente: **137 passed** sin cambios.

## Backwards compatibility

- **Agent flow identico**: `upsertTrip` sigue recibiendo los mismos parametros (`accountType='agent'`, `agencyId`, `tenantId`). Solo cambia el timing (debounce en vez de throttle). Los 137 tests pre-existentes no invocan el hook ‚Äî testean funciones puras.
- **Consumer flow identico**: `accountType='consumer'`, `ownerUserId` se derivan igual del AuthContext.
- **Error handling identico**: `upsertTrip(...).catch(() => {})` fire-and-forget, misma semantica que antes.

## Riesgos

- **R1 ‚Äî `beforeunload` + async `upsertTrip`**: El flush en `beforeunload` es sincronico e inicia el fetch. El browser permite que requests in-flight continuen brevemente. El upsert principal (la query critica) llega. Operaciones secundarias (`syncTripSegments`, CRM activity) pueden cortarse ‚Äî degradacion aceptable, `trips.planner_state` es lo que `loadPersistedPlannerState` lee.
- **R2 ‚Äî Browser crash sin `beforeunload`**: Ventana de perdida de 3s (vs 5s anterior). IndexedDB tiene el estado actualizado como fallback.
