# PR: Phase 1.1.a — B2C Ownership Schema Migration

## Que cambia

- Agrega `account_type` (`'agent'` | `'consumer'`) a `public.users` y `trips`, con CHECK XOR ligado al rol (`CONSUMER` enum value nuevo).
- Agrega `owner_user_id` (NOT NULL, FK a `users`) en `trips` con backfill via COALESCE(created_by, last_edited_by, sentinel).
- Agrega 3 consumer RLS policies en `trips` (`consumer_select_own_trips`, `consumer_insert_own_trips`, `consumer_update_own_trips`), expande el CHECK de `trips.status` con `'exploring'` y `'shared'`, y cambia FKs de `agency_id`/`tenant_id` de CASCADE a SET NULL.

## Por que dos migrations hermanas

PostgreSQL no permite usar un valor de enum recien agregado (`ALTER TYPE ADD VALUE`) en CHECKs, casts o comparaciones dentro de la misma transaccion. El nuevo valor solo es visible despues del commit.

- [`20260409000001_add_consumer_role_value.sql`](../../supabase/migrations/20260409000001_add_consumer_role_value.sql) — Solo `ALTER TYPE user_role ADD VALUE IF NOT EXISTS 'CONSUMER'`.
- [`20260409000002_b2c_ownership.sql`](../../supabase/migrations/20260409000002_b2c_ownership.sql) — Todo lo demas (15 secciones, ~290 lineas).

## Invariantes preservados

- **B2B intacto**: Todo row B2B existente sigue con `agency_id NOT NULL`, `tenant_id NOT NULL`, `account_type='agent'`.
- **Agent no puede crear trip consumer**: INSERT policy B2B reforzada con `AND trips.account_type = 'agent'`.
- **Consumer no puede crear trip agent**: Consumer INSERT policy exige `account_type = 'consumer'`.
- **Tests existentes verdes**: 129 passed, 7 skipped (mismo resultado pre y post migration).
- **Build TypeScript OK**: `tsc --noEmit` y `npm run build` sin errores.
- **Backfill completo**: 0 trips con `owner_user_id IS NULL`, 0 agent trips con `agency_id IS NULL`.
