# PR: Phase 1.1.e/f/g — Close Phase 1.1 (B2C trips cleanup)

## Scope

Cierra los tres sub-pasos que faltaban de la Fase 1.1 del roadmap B2C en una sola PR:

- **1.1.e** — Agrega `listTripsByUser(userId, accountType)` en `tripService.ts`, el read-side que le faltaba a B2C.
- **1.1.f** — Hace `deriveTripStatus` account-type-aware: los consumers con segments vacíos obtienen `'exploring'` en lugar de `'draft'`.
- **1.1.g** — Elimina el dual-write a `messages` con `meta.messageType='trip_planner_state'` en `persistPlannerState`. `trips` queda como único write path.

Continua [Fase 1.1.d](1.1.d-persist-debounce.md). Con este merge, la Fase 1.1 (B2C schema + adapter + SoT + debounce + read path + cleanup) queda cerrada.

## Problema que resuelve

### 1.1.e — Sin read path B2C
`tripService.ts` tenía `listTripsByAgency` para el caso B2B, pero no había forma de listar trips de un consumer individual. Sin esto no se puede construir la vista "Mis viajes" del consumer (parte del roadmap macro de la auditoría).

### 1.1.f — Estado inicial incorrecto para consumers
La migration 1.1.a (`20260409000002_b2c_ownership.sql` Sección 7) expandió el CHECK constraint de `trips.status` para incluir `'exploring'` y `'shared'`, pero `deriveTripStatus` seguía devolviendo `'draft'` para cualquier state con segments vacíos. Esto generaba trips de consumer con status `'draft'` — inconsistente con los RLS tests existentes (que usan `'exploring'` como seed) y con el lenguaje del producto companion-first.

### 1.1.g — Dual-write obsoleto
Desde 1.1.c, `trips` es source of truth para lectura. El write a `messages` con `meta.messageType='trip_planner_state'` ya no alimentaba ningún consumer legítimo del write-side. Solo existía por el fallback histórico del load path. Era código muerto en cada call a `persistPlannerState`.

## Que cambia

### `src/features/trip-planner/services/tripService.ts`

**1.1.e — nueva función `listTripsByUser`**:

```ts
export async function listTripsByUser(
  userId: string,
  accountType: 'agent' | 'consumer',
  filters?: { status?: string; limit?: number },
): Promise<TripRow[]>
```

- Filtra siempre por `owner_user_id = userId` + `account_type = accountType` + `neq('status', 'archived')`.
- Orden: `updated_at desc` (consistente con `listTripsByAgency`).
- Filtros opcionales mínimos: `status`, `limit`. Sin pagination offset, sin filtros de fecha (YAGNI — se agregan cuando la UI "Mis viajes" los pida).
- Shape de retorno: `TripRow[]` (mismo projection que `listTripsByAgency`).
- Error path: devuelve `[]` sin throw.

**Comportamiento por accountType**:
- `consumer`: las policies `consumer_select_own_trips` ya garantizan `owner_user_id = auth.uid()`. El filtro explícito es defense-in-depth.
- `agent`: devuelve "mis trips personales" (trips donde `owner_user_id === userId` y `account_type='agent'`). Es comportamiento **nuevo y complementario** a `listTripsByAgency`, no reemplazo.

**1.1.f — `deriveTripStatus` con `accountType`**:

```ts
function deriveTripStatus(
  state: TripPlannerState,
  accountType: 'agent' | 'consumer' = 'agent',
): string {
  const segments = state.segments || [];
  const initial = accountType === 'consumer' ? 'exploring' : 'draft';
  if (segments.length === 0) return initial;
  // 'quoted' y 'ready' se mantienen iguales para ambos accountTypes
  // ...
  return initial;
}
```

- Default `'agent'` preserva backwards compatibility.
- `upsertTrip` (línea 66) pasa `accountType` a `deriveTripStatus`.
- **`'shared'` no se deriva**: implica una acción explícita de compartir (tipo `shareTrip(tripId)`) que no existe aún. Si se hiciera derivable ahora sin campo en `plannerState`, sería código muerto. Se deja para una fase futura.

### `src/features/trip-planner/hooks/usePlannerState.ts`

**1.1.g — eliminación del dual-write**:

- Borradas las 27 líneas (`delete` + `insert` de `supabase.from('messages')` con `meta.messageType='trip_planner_state'`, más sus bloques de error handling).
- `persistPlannerState` post-cambio: normaliza state → escribe IndexedDB fire-and-forget → schedule del debounced upsert a `trips`. Fin.

## Que NO se toco

- **`listTripsByAgency`** — B2B intacto. Cualquier consumer del service de B2B (Reports, CRM, Dashboard de agency) sigue funcionando igual.
- **`loadPersistedPlannerState`** — cerrado en 1.1.c. Su fallback chain `IndexedDB → trips → messages` se mantiene intacta. Las conversaciones legacy pre-1.1.b que nunca tuvieron un trip row siguen leyéndose correctamente desde `messages`. Solo las **nuevas** escrituras dejan de ir a `messages`; las rows existentes siguen ahí y son readables.
- **`persistPlannerState` debounce/flush** — cerrado en 1.1.d. El scheduler, el flush on unmount/beforeunload, y el flush en cambio de conversación se mantienen bit-a-bit.
- **`upsertTrip` signature y payload** — idénticos. Solo cambia que `deriveTripStatus` recibe el `accountType` que ya tenía en el scope.
- **`duplicateTrip`, `updateTripLeadId`, `getTripByConversation`, `getTripById`** — sin cambios.
- **`AuthContext`** — ya exponía `accountType` desde 1.1.b.
- UI, componentes, rutas, layouts, orchestrator, `useMessageHandler`, `planner_agent`.
- Schema de base de datos — sin migrations. Código puro.

## Tests

### 5 unit tests nuevos en `listTripsByUser.test.ts`

1. **Consumer**: la query aplica `.eq('owner_user_id', userId)`, `.eq('account_type', 'consumer')`, `.neq('status', 'archived')`, `.order('updated_at', { ascending: false })`.
2. **Agent**: mismo pattern con `account_type='agent'`.
3. **`filters.status`**: aplica `.eq('status', ...)` adicional cuando se pasa.
4. **`filters.limit`**: aplica `.limit(n)` cuando se pasa.
5. **Error path**: devuelve `[]` cuando supabase devuelve `{ data: null, error }`.

Mock thenable chain-style: un único builder que implementa todos los métodos chainable y expone `.then()` para terminar la query. Más limpio que mocks individuales por método.

### 2 unit tests nuevos en `upsertTripAdapter.test.ts` (caso 7 y 8)

- **Caso 7**: `upsertTrip` con `accountType='consumer'` y segments vacíos → payload.status === `'exploring'`.
- **Caso 8**: `upsertTrip` con `accountType='agent'` y segments vacíos → payload.status === `'draft'` (B2B regression).

Los 6 casos pre-existentes del archivo siguen verdes sin modificación porque ninguno assertea sobre `status` directamente, y el default param de `deriveTripStatus` los protege.

### 2 tests RLS integration en `b2cOwnershipRls.test.ts` (test 10 y 11)

- **Test 10**: `consumerClient` replica manualmente la query de `listTripsByUser(consumerUserId, 'consumer')` → ve su propio trip, no el de Consumer B, no el del agent.
- **Test 11**: `agentClient` replica la query con `accountType='agent'` → ve su agent trip.

Se replican manualmente con los clients autenticados del setup en lugar de importar la service function porque `listTripsByUser` usa el cliente global `@/integrations/supabase/client`, que en entorno Node tira `localStorage is not defined` (D11). La mecánica del query (RLS + filtros explícitos) queda cubierta idénticamente.

Ambos tests están dentro de `describe.skipIf(!canRun)` y requieren `SUPABASE_SERVICE_ROLE_KEY`. Sin key: se suman al skipped count (14 en total). Con key: corren como parte del suite RLS expandido a 11 tests.

## Backwards compatibility

- **B2B sin cambios observables**. `listTripsByAgency` intacta. `deriveTripStatus` con default `'agent'` preserva el comportamiento. `upsertTrip` no cambió su signature ni su payload para agents. Los 6 tests pre-existentes de `upsertTripAdapter.test.ts` siguen verdes sin tocarlos.
- **Consumer trips pre-existentes** (creados desde 1.1.b con status `'draft'`) quedan en el DB como `'draft'`. El nuevo código los "arreglará" automáticamente: al siguiente edit, el debounce dispara un upsert que escribe `'exploring'` (si los segments siguen vacíos). Sin migration retroactiva — innecesaria.
- **`loadPersistedPlannerState`** sigue leyendo de `messages` como fallback para conversaciones legacy pre-1.1.b sin trip row. El read path no se tocó. Las rows existentes en `messages` con `meta.messageType='trip_planner_state'` siguen siendo readable; solo no se escriben nuevas.

## Riesgos

- **R1 — Agent histórico con `owner_user_id` ≠ `created_by`**: El backfill de 1.1.a usó `COALESCE(created_by, last_edited_by, sentinel)` (migration Sección 4). Para agents con `created_by` válido (el caso común), `owner_user_id === created_by`. El filtro nuevo `owner_user_id = userId` va a matchear los trips que el agent creó personalmente — que es semánticamente correcto para "mis trips personales". Los trips creados por otros agents de la misma agency no aparecen en `listTripsByUser(agent, 'agent')` — correcto, para eso sigue existiendo `listTripsByAgency`.
- **R2 — Crash duro sin beforeunload en el primer edit**: Hereda D17 de 1.1.d. Sin cambio: este PR no lo empeora ni lo mejora. IndexedDB sigue siendo el safety net local sincrónico.
- **R3 — Conversación legacy sin trip row después de 1.1.g**: El load path mantiene la fallback chain (`cache → trips → messages`). Legacy sigue funcionando. Al primer edit post-merge, el debounce upserta un trip row, y todos los loads subsiguientes preferirán `trips`.
